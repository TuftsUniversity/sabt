The TAPER web application

The TAPER web application serves the following purposes:

  • Gives endusers a friendly, wizard-ish interface for creating new RSA documents
  • Gives staff an administrative interface for approving and finishing these documents (and also alerts the staffwhen new RSAs are awaiting approval)
  • Allows staff to create and maintain SSA documents
  • Allows staff to control access to the TAPER web application.

The first three tasks make up the part of the application known as the Submission Agreement Builder Tool (SABT).

Location

All the code, configuration files, and templates that make up the web application are in the TAPER Subversion repository, in the directory TAPER/. The contents are organized in a fashion typical for a Catalyst-based web application.

Front end

Templates

The templates that define the web pages' content are in TAPER/root/src. Per Catalyst best practices, they are Template Toolkit files, and are intended to be freely editable by non-coders, as necessary. (Though non-coders will probably wish to avoid making any changes to any template's TT tags.)

More on Template Toolkit: http://template-toolkit.org/

Forms

Per Catalyst best practices, many of the TAPER web application's forms are defined by YAML configuration files which are processed by the app's controller modules, by way of the Perl HTML::FormFu CPAN module (c.f.). They may be found in TAPER/root/forms.

Non-coders can, to a certain extent, modify these files in order to customize things like error message text, or form labels, or class attributes (for CSS styling purposes). Be aware that changing other information, such as the form fields' names or ID attributes, can cause them to not work any more (unless, of course, you are a coder, and make appropriate changes to the corresponding controller code).

CSS

The application's CSS file is in TAPER/root/static/css.

Backend

The TAPER web app hangs on the Catalyst framework, and is constructed with an adherence to Catalyst best practices, so that general reference material about Catalyst should apply to the TAPER code. Please see TAPER software nuts and bolts for more information about its gory details, and pointers to further reading.

The namespace that its Perl-based backend uses is TAPER. Thus, its sub-components start with prefixes like TAPER::Controller, TAPER::Schema, et cetera.

Authentication

To authenticate with the TAPER web application, two things must be true about the user:

1. They must have an account on the LDAP system defined in taper.conf or taper_local.conf.
2. They must have an account with the TAPER web application.

Both are needed because the TAPER app doesn't store any password information, instead relying entirely on LDAP for login-time authentication.

After the user authenticates with LDAP, the app makes sure that the user also has representation within TAPER's own database. If they do, they're welcomed into the application.

Developer documentation

You can find the web app's developer documentation attached to the Perl modules that define its backend. Every module that bears an API ends with a section of POD (inline Perl) documentation, which you can read either by opening that module in a text editor or (more likely) feeding it to the perldoc command-line program.

Start your reading with lib/TAPER.pm, which should contain adequate pointers for the other modules, and their associated documentation.
